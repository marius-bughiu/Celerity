All notable changes to Celerity are documented here. This project follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/) and uses [Semantic Versioning](https://semver.org/spec/v2.0.0.html). Version numbers are produced by MinVer from the `v`-prefixed git tags.

## [Unreleased]

### Changed

- The default hasher on the `LongDictionary<TValue>` and `LongSet` convenience subclasses switched from `Int64WangHasher` (the 11-op Thomas-Wang finalizer) to the new `Int64WangNaiveHasher` (a 2-op XOR-fold of `(int)key ^ (int)(key >>> 32) ^ (int)(key >>> 16)`). The previous default was paying roughly 5x the per-call hash cost of the naive variant on every `Hash`, and that cost compounds across the resize chain (every entry is re-hashed on resize) and across the backward-shift cluster scans (every scanned slot is re-hashed for the cyclic comparison) — the live benchmark dashboard was showing the `Long*` collections losing to BCL across every shape (1.62x on `LongDict.Insert(100k)`, 1.48x on `LongSet.Add(100k)`, 1.29-1.32x on `LongDict` / `LongSet.Remove`) almost entirely because of that. The middle-16-bits fold in the new default keeps enough high-half entropy alive that keys whose low 32 bits are sequential but upper bits carry structure (type tags, shard IDs, monotonically allocated IDs) still distribute reasonably — `Int64WangNaiveHasher` is intended as the new common-case fast path, not as a uniform replacement. **This is a user-visible behavioural change** on the parameterless `new LongDictionary<TValue>()` / `new LongSet()` constructors: callers who picked the convenience subclasses for adversarial-key collision behaviour will silently get an XOR-fold instead of the full Wang finalizer. Opt back into the previous default with the open-generic shape: `new LongDictionary<TValue, Int64WangHasher>()` / `new LongSet<Int64WangHasher>()`, or escalate further to `Int64Murmur3Hasher` for better avalanche under adversarial workloads. `Int64WangHasher` stays in the box and remains the right answer for those workloads; the choosing-a-hasher table in `docs/api/hashing.md` documents the new selection axis. Pinned by the new `Int64WangNaiveHasherTests` coverage (theory inputs across zero / `1` / `-1` / `42` / `long.MaxValue` / `long.MinValue` / `1234567890123456789L` / 48-bit-set / alternating-nibble keys, determinism across calls and struct instances, high-bit and mid-bit influence regression guards, consecutive-small-input and single-high-bit-strided distinctness sweeps) plus the existing 16-file `LongDictionary` / `LongSet` test surface running green against the new default. Closes #108.
- `BackwardShiftRemove` replaces `RehashAfterRemove` on `IntDictionary<TValue, THasher>`, `LongDictionary<TValue, THasher>`, `CelerityDictionary<TKey, TValue, THasher>`, `IntSet<THasher>`, `LongSet<THasher>`, and `CeleritySet<T, THasher>` — the new algorithm is Knuth TAOCP Vol 3, §6.4 Algorithm R. The previous `RehashAfterRemove` walked the cluster forward and, for every surviving entry, re-hashed the key and re-probed from its natural slot for an empty target; the inner probe was a second pass over the cluster, making the work-per-cluster quadratic where it should be linear. The dashboard was showing `Remove` losing to BCL across every collection, badly at small `N` (up to 5.10x on `CeleritySet_Remove(1k)` and 2.46x on `IntSet_Remove(1k)`) because of this. Backward-shift visits each surviving cluster entry exactly once and only moves it when the cyclic invariant says it must — for the gap at `i` and the scan position `j` with natural slot `k`: if `i <= j` bypass when `i < k && k <= j`, otherwise (`i > j`, the scan has wrapped) bypass when `i < k || k <= j`. The `Remove` call site no longer pre-clears the slot; the new helper writes the final empty entry itself, saving a redundant store on the common path where shifts collapse the gap. `Count`, the out-of-band zero / default key, `_version`-based enumerator invalidation, and the contiguous-cluster invariant under linear probing are all preserved. Pinned by one new wrap-around regression test per collection driven by a new `IdentityIntHasher` / `IdentityLongHasher` (cluster crossing the array boundary, mixed natural slots — exercises all three branches of the cyclic comparison), plus the existing `ResizeThenRemoveSweep_*_UnderFullCollision` facts continuing to pin the long-chain behaviour after multiple resizes.
- `[MethodImpl(MethodImplOptions.AggressiveInlining)]` is now applied to every `ProbeForInsert` / `ProbeForKey` / `ProbeForItem` / `BackwardShiftRemove` on all six collections (`IntDictionary`, `LongDictionary`, `CelerityDictionary`, `IntSet`, `LongSet`, `CeleritySet`). The hashers themselves already carried the attribute, but the methods that called them did not — every probe-driven entry point (`TryAdd`, the indexer get/set, `Remove`, `Contains`) crossed a function boundary on every operation even though the probe + shift methods are short and called from a small number of sites. With the hint, the JIT collapses the `probe → hash → mask → compare` pipeline into the caller's stack frame and folds the loop body into a tight `mov` + `and` + `cmp` + `je` sequence. Mechanical change — every behavioural and surface contract is unchanged.
- The probe and shift hot paths on all six collections now read `_keys[index]` / `_slots[index]` via `Unsafe.Add` against `MemoryMarshal.GetArrayDataReference` instead of the indexer, eliminating the per-iteration bounds check on every probe walk and every shift step. Sites updated: `ProbeForInsert` + `ProbeForKey` + `BackwardShiftRemove` + the indexer setter writes + the `TryAdd` writes on the dictionaries; `ProbeForItem` + the inline probe loop in `TryAdd` + `BackwardShiftRemove` + the `TryAdd` write on the sets. The safety invariant is structural: every collection has `mask = _keys.Length - 1` and `index = ... & mask`, so `index ∈ [0, _keys.Length)` is guaranteed for every iteration. Casts use the `(nint)(uint)` pattern so the JIT widens via zero-extension on x64 and gets the non-negativity hint that unlocks the tightest emit. The collision suites (every `ConstantIntHasher` / `ConstantLongHasher` fact plus the new `IdentityIntHasher` / `IdentityLongHasher` wrap-around facts) drive the probe loop right up against the cyclic boundary and would fault loudly on any off-by-one. Lookup-only paths (`ContainsKey`, `TryGetValue`, indexer getter, `Contains`) were already winning vs BCL and were left to a follow-up; that follow-up is tracked in PR #106.
- `ProbeForInsert` on `IntDictionary<TValue, THasher>`, `LongDictionary<TValue, THasher>`, and `CelerityDictionary<TKey, TValue, THasher>` now returns a `bool wasEmpty` out-parameter so the indexer setter and `TryAdd` no longer have to re-read `_keys[index]` after the probe to decide between bump-`_count` and overwrite. The probe loop already knows whether it exited on an empty slot or on a key match; lifting that signal to the caller saves one bounds-checked array load per insert (and one `EqualityComparer<TKey>.Default` dispatch on `CelerityDictionary`) on the hot insert path. The `_keys` hoist + single `mask` computation at the top of `ProbeForInsert` is now shared between the wrap step and the BCE arithmetic. The sets inline their probe loop in `TryAdd` already, so there is no separate `ProbeForInsert` to refactor on the set side. Pinned by `IntDictionary` / `LongDictionary` / `CelerityDictionary` `_Add_ThenIndexerOverwrite_ShouldSucceed` facts and the new `Dictionaries_RepeatedIndexerOverwrite_ShouldKeepCountStable` 100-round stress test across all three shapes (catches a wrong-branch wire-up in the `wasEmpty` signal that would only surface under aggregate workloads).
- `Resize()` on `IntDictionary<TValue, THasher>`, `IntSet<THasher>`, `LongDictionary<TValue, THasher>`, `LongSet<THasher>`, `CelerityDictionary<TKey, TValue, THasher>`, and `CeleritySet<T, THasher>` now reads from the old arrays and probes / writes into the new arrays via `Unsafe.Add` against `MemoryMarshal.GetArrayDataReference` instead of the indexer. This applies the same bounds-check-elimination pattern already used on `ProbeForInsert` / `ProbeForKey` / `BackwardShiftRemove` to the resize migration: every iteration's `oldKeys[i]` / `oldSlots[i]` load, the probe-for-empty-slot walk on the new arrays, and the final write all dropped their per-iteration `Buffer.GetMaximumArrayLength` / cmp-against-`Length` bounds checks. The bounds are structural — `i < oldKeys.Length` is the for-loop condition, and `index = ... & mask` keeps `index ∈ [0, newSize)` because `mask = newSize - 1` and `newSize` is a power of two — so the per-element BCE removal is provably safe. Casts use the `(nint)(uint)` pattern to give the JIT the non-negativity hint that unlocks a zero-extending load on x64. No public-API change; correctness is unchanged. Pinned by the existing resize-heavy coverage (`LoadFactorBoundaryTests`, `TryAddDuplicateResizeTests`, the `ResizeThenRemoveSweep_*_UnderFullCollision` Facts in all four `*CollisionTests` files, and the multi-resize-from-tiny-capacity Facts across all six collections) — all 746 tests pass locally on `net8.0`.

### Added

- `Int64WangNaiveHasher` in `Celerity.Hashing` — lightweight XOR-fold hash provider for `long` keys, the 64-bit counterpart to `Int32WangNaiveHasher`. Struct hasher, `[MethodImpl(MethodImplOptions.AggressiveInlining)]`. Two ops vs eleven for the full Thomas-Wang finalizer: `(int)key ^ (int)((ulong)key >> 32) ^ (int)((ulong)key >> 16)`. The extra middle-16-bits fold over a naive `(int)key.GetHashCode()` keeps a chunk of the high-half entropy in the result, which materially improves distribution on keys whose low 32 bits are sequential (e.g. monotonically allocated IDs whose upper bits carry type / shard). Now the default hasher for the `LongDictionary<TValue>` and `LongSet` convenience subclasses — see the `### Changed` entry above for the user-visible default-hasher switch and how to opt back into the previous default. Prefer it when the key distribution is already reasonably uniform and latency matters more than collision resistance; for adversarial or heavily clustered keys, escalate to `Int64WangHasher` (full Wang finalizer) or `Int64Murmur3Hasher`.
- `LongSet<THasher>` and the `LongSet` convenience subclass — high-performance set of `long` values, mirroring the `IntSet` surface for 64-bit elements. Defaults to `Int64WangNaiveHasher` (matching `LongDictionary`'s default after the default-hasher switch noted in the `### Changed` entry above). Implements `IEnumerable<long>`, ships an allocation-free struct enumerator with `_version`-based concurrent-modification detection, the `IEnumerable<long>` constructor with `ICollection<long>` count sizing and silent dedupe per BCL `HashSet<T>` semantics, full `Add` / `TryAdd` / `Contains` / `Remove` / `Clear` surface, and constructor validation matching `IntSet`. The zero element (`0L`) collides with the `EMPTY_SLOT` sentinel exactly as on `IntSet` and is stored out-of-band via a dedicated flag, so `set.Add(0L)` is round-trippable. Closes the last remaining dictionary-to-set parity gap (`IntDictionary` → `IntSet`, `CelerityDictionary` → `CeleritySet`, `LongDictionary` → `LongSet`).
- `StringMurmur3Hasher` in `Celerity.Hashing` — MurmurHash3 x86_32 over a string's UTF-16 code units. Struct hasher, `AggressiveInlining`. This is the `string` counterpart to `Int32Murmur3Hasher` / `Int64Murmur3Hasher`: it gives strings the same strong-escalation option that `int` and `long` already have over their fast-fold defaults, closing the gap where `string` shipped only `StringFnV1AHasher`. It improves on `StringFnV1AHasher` in two ways. First, it consumes the full 16-bit value of every character (reading consecutive char pairs as little-endian 32-bit blocks, exactly as a byte-oriented Murmur3 would over the native UTF-16 byte stream), so it distinguishes characters that share a low byte but differ in their upper byte — e.g. `'A'` (`U+0041`) and `'Ł'` (`U+0141`), which `StringFnV1AHasher` (low-byte only) collides on. Second, the `fmix32` finalizer gives every input bit influence over every output bit, holding distribution up on clustered or adversarial keys. The empty string `""` maps to `0` (the fixed point of `fmix32` over a zero accumulator), mirroring the `0 → 0` fixed point of the integer Murmur3 hashers; the dictionaries route the out-of-band `null`-key entry around the hasher so this never collides with the empty-slot sentinel. `Hash(null)` throws `ArgumentNullException` (parameter `"key"`), matching `StringFnV1AHasher`. Prefer it over `StringFnV1AHasher` for keys with significant non-ASCII content or when collision resistance matters more than the few cycles FNV-1a saves on short ASCII keys. Incremental progress on issue #24.
- `StringMurmur3HasherTests` — empty-string `→ 0` and five additional exact-value anchors (`"a"`, `"ab"`, `"abc"`, `"hello"`, `"hello world"`) that pin the algorithm against block-read-order / tail-handling / finalizer-constant drift; determinism across calls and across stateless struct instances; case sensitivity; single-character-change and append-character avalanche; the headline full-character-distinction regression (`StringFnV1AHasher` collides `"A"` / `"Ł"` and `"AB"` / `"Łł"`, `StringMurmur3Hasher` keeps them distinct); a 1000-distinct-string collision-free sweep; `Hash(null)` throwing `ArgumentNullException`; and integration tests driving `CelerityDictionary<string, …, StringMurmur3Hasher>` and `CeleritySet<string, StringMurmur3Hasher>`, including the empty-string regular key, a non-ASCII key, and the out-of-band `null` key.
- `docs/api/hashing.md` and `README.md` — a full `StringMurmur3Hasher` section (algorithm, when to prefer it over `StringFnV1AHasher`, empty-string `→ 0` note) added next to the existing `StringFnV1AHasher` entry, the choosing-a-hasher table's `string` row updated to list `StringMurmur3Hasher` as the escalation option, the `StringFnV1AHasher` low-byte note now points at it, and a `string` bullet added to the README hasher-picking notes.
- `LongSetTests` — mirrors the `IntSetTests` coverage (CRUD, zero element, resize survival, remove-then-reinsert, `Clear` reuse, constructor validation, `IEnumerable<long>` ctor null/dedupe, struct-enumerator smoke test, enumerator-invalidation on mutation, `LongSet<THasher>` open-generic smoke) plus long-specific cases: extreme element values (`long.MaxValue`, `long.MinValue`, `int.MaxValue + 1L`, `int.MinValue - 1L`, `-1L`) and a regression test that two elements sharing the same lower 32 bits but differing in the upper 32 bits are kept distinct (guards against any accidental int-truncation on the probe path).

## [1.2.1] - 2026-05-17

### Fixed

- `TryAdd` (and therefore `Add`) on `IntDictionary<TValue, THasher>`, `LongDictionary<TValue, THasher>`, `CelerityDictionary<TKey, TValue, THasher>`, `IntSet<THasher>`, and `CeleritySet<T, THasher>` no longer calls `Resize()` on the duplicate-key path. Previously, when the collection was exactly at the load-factor threshold and the caller supplied a duplicate key, `TryAdd` would hit the `if (_count >= _threshold) Resize();` check before the duplicate check, swap out the backing arrays, then walk the probe chain in the new table, detect the duplicate, and return `false` — all without bumping `_version`. An enumerator captured before that call would pass its `_version` check on the next `MoveNext` (because `_version` was unchanged) and silently read from the new arrays with an `_index` from the old coordinate system, yielding stale / duplicated / missing entries instead of throwing `InvalidOperationException`. This was a regression from the single-probe `TryAdd` rewrite in PR #53 — pre-#53 `TryAdd` was `if (ContainsKey(key)) return false; this[key] = value;` and `ContainsKey` does not mutate, so `Resize` only ran on the actual insert path (which bumps `_version`). The fix reorders all five `TryAdd` implementations so the duplicate-detection probe runs *before* the threshold check: a duplicate-at-threshold call now returns `false` without touching `_count`, `_version`, or the backing arrays, and a new-key-at-threshold call probes once in the current table to confirm the key is new, then `Resize()`s and re-probes in the doubled table to find an insert slot. Active enumerator invalidation now follows the visible-mutation rule again: `_version` bumps if and only if the collection actually changed. Pinned by `TryAddDuplicateResizeTests`, which on each of the five collections constructs the collection at exactly `capacity=4` (threshold = 3), fills it to threshold, captures an enumerator, and asserts that a duplicate `TryAdd` / `Add` leaves the enumerator valid (it can complete the enumeration) while a new-key `TryAdd` invalidates it. Closes #92.

### Changed

- `docs/api/collections.md` now covers the public API additions that shipped in `[1.2.0]` but were never propagated into the API reference: `ContainsValue(TValue? value)` on `CelerityDictionary`, `IntDictionary`, and `LongDictionary`, and the `IEnumerable<T>` constructor on `CeleritySet<T, THasher>`, `IntSet<THasher>`, and the `IntSet` convenience subclass. `ContainsValue` is added as a full subsection under `CelerityDictionary` (between `TryGetValue` and `Add`, mirroring source order) and as a bullet on the shared `IntDictionary` / `LongDictionary` method lists. The set `IEnumerable<T>` ctor is documented alongside the existing `(capacity, loadFactor)` ctor on all three set types, including an explicit note that — unlike the dictionary `IEnumerable<KeyValuePair<,>>` ctor — duplicate elements are silently deduplicated to match BCL `HashSet<T>` semantics. `IntSet<THasher>` previously had no Constructors subsection at all; one is now added. No code or behavior changes — `docs/api/collections.md` is the authoritative API-reference target linked from `README.md`, and this closes the drift gap so callers landing there to look up these methods see the actual shipped surface. Closes #89.
- `CelerityDictionary<TKey, TValue, THasher>`'s primary indexer get/set now uses `TValue` instead of `TValue?`, matching the signature `IntDictionary<TValue, THasher>` and `LongDictionary<TValue, THasher>` already shipped. The indexer never returns null as a "miss" sentinel — a missing key throws `KeyNotFoundException` — so the previous `TValue?` annotation was misleading and gave callers `string s = celerityDict[key];`-style CS8600 nullable warnings that the parallel `IntDictionary` / `LongDictionary` callers did not get. The interior reads from the backing `_values` and `_defaultKeyValue` slots are unchanged; they just acquire the same null-forgiving `!` that `IntDictionary` already uses on its `_values[index]!` read. The `IReadOnlyDictionary<TKey, TValue?>` interface contract is preserved through a new explicit interface forwarder (`TValue? IReadOnlyDictionary<TKey, TValue?>.this[TKey key] => this[key];`), mirroring how `IntDictionary` and `LongDictionary` already widen their non-nullable primary indexer to the nullable interface signature. The `docs/api/collections.md` indexer signature is updated to match. Source-compatible at the binary level (nullability is metadata-only) and at the source level for non-null call sites; the only behavioural change is that callers no longer have to suppress a CS8600 to assign the indexer result to a non-nullable local. Pinned by `IndexerReturnTypeTests`, which reflects on each dictionary's public indexer and asserts `PropertyType == TValue`, plus three compile-time tests that assign `dict[k]` directly to a non-nullable `string` local on all three dictionary shapes. Closes #88.
- `IntDictionary<TValue, THasher>.Resize()` / `RehashAfterRemove()`, `CelerityDictionary<TKey, TValue, THasher>.Resize()` / `RehashAfterRemove()`, `IntSet<THasher>.Resize()` / `RehashAfterRemove()`, and `CeleritySet<T, THasher>.Resize()` / `RehashAfterRemove()` now reinsert entries directly into the backing arrays instead of routing through the public indexer setter / `InsertNonZero` / `InsertNonDefault` helpers — applying the same tightened migration shape that `LongDictionary` got in PR #84 to the remaining four collections. The bypassed helpers perform an equality check inside the probe walk (looking for an existing entry to overwrite), test `_count >= _threshold`, branch on the empty-slot sentinel, and bump `_count` and `_version` per call — none of which is meaningful during a migration: every reinserted key is unique in the destination, the threshold can't trigger, the sentinel was already filtered, and `_count` / `_version` are conserved across the migration. `Resize` now builds into a fresh local pair of arrays and swaps them in at the end, which also keeps the collection in a consistent state if the migration ever throws. The now-unused `IntSet.InsertNonZero` and `CeleritySet.InsertNonDefault` private helpers are removed. Behaviour is unchanged — `Count`, `_version`-based enumerator invalidation, the zero / default out-of-band slots, and the contiguous-cluster invariant under linear probing are all preserved. Pinned by new `ResizeThenRemoveSweep_*_UnderFullCollision` Facts in `IntDictionaryCollisionTests`, `CelerityDictionaryCollisionTests`, `IntSetCollisionTests`, and `CeleritySetCollisionTests`, each running 40-entry insert past threshold + every-other remove + reinsert under maximum collision pressure. Closes #83.
- `LongDictionary<TValue, THasher>.Resize()` and `RehashAfterRemove()` now reinsert entries directly into the backing arrays instead of routing through the public indexer setter. The indexer setter is designed for user-facing inserts: it performs an equality check inside `ProbeForInsert` (looking for an existing entry to overwrite), tests `_count >= _threshold`, branches on the zero-key sentinel, and bumps `_count` and `_version` per call. None of that work is meaningful during a migration — the key being reinserted is guaranteed to be unique in the destination, the threshold can't trigger (we just doubled it during `Resize`, or just decremented `_count` during `RehashAfterRemove`), the zero-key entry is out-of-band and filtered before the call, and `_count` / `_version` are conserved across the migration so the user-visible operation that triggered it (e.g. `TryAdd` → `Resize`) bumps `_version` exactly once instead of `N + 1` times. `Resize` now builds into a fresh local pair of arrays and swaps them in at the end, which also keeps the dictionary in a consistent state if the migration ever throws. Behaviour is unchanged — `Count`, `_version`-based enumerator invalidation, the zero-key out-of-band slot, and the contiguous-cluster invariant under linear probing are all preserved. The same anti-pattern still lives in `IntDictionary`, `CelerityDictionary`, `IntSet`, and `CeleritySet`; a follow-up issue will track applying the same fix there. Pinned by the new `LongDictionaryCollisionTests.ResizeThenRemoveSweep_ShouldPreserveAllRemainingKeys_UnderFullCollision` Fact (40-key insert past threshold + every-other remove + reinsert under maximum collision pressure). Closes issue #82.
- `LongDictionary<TValue, THasher>.TryAdd` now walks the probe chain exactly **once** per call instead of twice, matching the single-probe rewrite that PR #53 applied to `IntDictionary`, `CelerityDictionary`, `IntSet`, and `CeleritySet`. `LongDictionary` was added in the same release as that rewrite but landed after it in commit order, so it was missed: `TryAdd` was still calling `ContainsKey` followed by the indexer setter, doing two `ProbeForInsert`-style walks per call. Behaviour is unchanged — `Add` still throws on duplicates, `TryAdd` still returns `false` and leaves the existing value untouched, the out-of-band zero-key path is unchanged — but bulk-loads via the `IEnumerable<KeyValuePair<long, TValue>>` constructor and any `if (!dict.TryAdd(...))` insert-or-skip pattern now do roughly half the probe work. Pinned by three new `LongDictionary`-shaped Facts in `TryAddProbeCountTests` (`_NewKey_DoesExactlyOneProbeWalk`, `_DuplicateKey_DoesExactlyOneProbeWalk`, `_PreservesExistingValueOnDuplicate`) using a counting `IHashProvider<long>` mirror of the existing `CountingIntHasher`. Closes issue #77.

### Fixed

- The `IEnumerable` constructor on `IntDictionary<TValue, THasher>`, `LongDictionary<TValue, THasher>`, `CelerityDictionary<TKey, TValue, THasher>`, `IntSet<THasher>`, and `CeleritySet<T, THasher>` (and the `IntDictionary<TValue>` / `LongDictionary<TValue>` / `IntSet` convenience subclasses) now surfaces a `null` source as `ArgumentNullException` even when the caller also passed an invalid `loadFactor` or `capacity`. Previously, the null check sat in the constructor body *after* the chained `: this(Math.Max(capacity, (source as ICollection<...>)?.Count ?? 0), loadFactor)` call: that chain forwarded the user-supplied `loadFactor` into the primary constructor's validation, which threw `ArgumentOutOfRangeException("loadFactor", ...)` first when the load factor was outside `(0, 1)`, masking the more important null-argument violation. The null check is now lifted into a private static `InitialCapacityForSource` helper invoked as part of the chained-ctor argument expression, so it runs before the primary ctor's range validation and matches the framework design guideline that `ArgumentNullException` for a parameter should be reported before other validation errors on sibling parameters. Behaviour is unchanged on the happy path and unchanged for any single-argument failure (a non-null source with bad `loadFactor` still throws `ArgumentOutOfRangeException`); only the null-source-plus-other-invalid-argument path is corrected. Pinned by the new `IEnumerableConstructorNullPriorityTests` (one Fact per collection, plus an `IntDictionary` boundary Fact for `loadFactor: 0f` and a generic-shape Fact for `IntDictionary<,>`, `LongDictionary<,>`, `IntSet<>`). Closes #94.
- `StringFnV1AHasher.Hash(null)` now throws `ArgumentNullException` (parameter name `"key"`) instead of `NullReferenceException`. Public APIs should signal a null argument as an explicit contract violation, not as an unchecked dereference. The Celerity dictionaries store the out-of-band `null` / `default(TKey)` key entry without ever calling the hasher, so the surface area of this change is limited to direct `StringFnV1AHasher` usage and to consumers that plug the hasher into custom `IHashProvider<string>` callers that do not handle the null-key slot themselves. The XML doc comment on `Hash` now declares the exception, and `StringFnV1AHasherTests.Hash_NullString_*` is updated to assert `ArgumentNullException` rather than pinning the previous wart. Closes #71.

### Fixed

- **`IntDictionary<TValue>` constructor arguments were silently discarded.** The convenience subclass `IntDictionary<TValue>` accepted `capacity` and `loadFactor` parameters but forwarded to `: base()` with no arguments, so every instance was created with the defaults regardless of what the caller passed. It now forwards `capacity` and `loadFactor` to the base constructor.
- **`IntDictionary` could not store the key `0`.** `EMPTY_KEY = 0` was used as the "empty slot" sentinel, which collided with the legitimate key value `0`. `map[0] = x` appeared to succeed but subsequent `ContainsKey(0)`, `map[0]`, and `Count` returned wrong answers. The zero key is now stored out-of-band via a dedicated flag + value slot, a pattern borrowed from `fastutil` / `HPPC`.
- **`CelerityDictionary` could not store `default(TKey)`.** Same root cause as above, generalized: `default(int)` / `default(long)` / `default(Guid)` / `null` strings were all lost. Fixed the same way, via a `_hasDefaultKey` flag and a dedicated value slot.

- Constructor validation test suite (`ConstructorValidationTests.cs`) covering rejection of invalid `loadFactor` (≤0, ≥1) and negative `capacity` for both `IntDictionary` and `CelerityDictionary`, plus acceptance of valid edge values.

### Fixed (additional)

- **`CelerityDictionary` and `IntDictionary` accepted invalid constructor arguments.** `loadFactor >= 1.0` caused an infinite loop in `ProbeForInsert` once the table was full; `loadFactor <= 0` caused a resize on every insert. Both constructors now throw `ArgumentOutOfRangeException` for `capacity < 0`, `loadFactor <= 0`, or `loadFactor >= 1`.

### Changed

- `TryAdd` (and therefore `Add`) on `IntDictionary<TValue, THasher>`, `CelerityDictionary<TKey, TValue, THasher>`, `IntSet<THasher>`, and `CeleritySet<T, THasher>` now walks the probe chain exactly **once** per call instead of twice. The previous implementation called `ContainsKey` / `Contains` followed by the indexer setter / `InsertNon*` helper, each starting its own probe walk; the rewrite uses a single `ProbeForInsert`-style walk that either lands on the existing entry (return `false`) or on the first empty slot (insert in place). Behaviour is identical to before — including the duplicate-key contract on `Add` and the "unchanged on duplicate" contract on `TryAdd` — but bulk-loads via the new `IEnumerable<KeyValuePair<,>>` constructor and any `Add`-heavy hot path now do roughly half the probe work. Closes issue #24. Pinned by `TryAddProbeCountTests`, which uses a counting `IHashProvider` to assert that `TryAdd` calls `Hash` exactly once on both the new-key and duplicate-key paths across all four collections.
- The `IntDictionary` `EMPTY_VALUE` field is now `static readonly` instead of an instance field. No behavior change; just removes per-instance overhead.
