### Added

- `StringFnV1Hasher` in `Celerity.Hashing` — the original **FNV-1** 32-bit hash provider (multiply-then-XOR) for `string` keys, over the full little-endian UTF-16 byte stream (both bytes of every character). Struct hasher, `[MethodImpl(MethodImplOptions.AggressiveInlining)]`. It is the FNV-1 counterpart to `StringFnV1AFullHasher`: FNV-1 and FNV-1a share the same constants (offset basis `2166136261`, prime `16777619`) and differ only in the per-byte step order — FNV-1 multiplies first then XORs the byte (`hash = hash * prime; hash ^= b`), where FNV-1a XORs first then multiplies. That ordering is the whole story behind the family: in FNV-1 the last byte folded is only XORed in with no following multiply to diffuse it, so a change in the trailing byte avalanches less thoroughly than in FNV-1a, whose final operation is always a multiply — which is exactly why FNV-1a became the generally preferred default. `StringFnV1Hasher` is provided for users who specifically need the original FNV-1 ordering (e.g. to match an external system that hashes with FNV-1). Like the other full-width string hashers it folds the full 16-bit value of every character (low byte then high byte), so unlike the low-byte `StringFnV1AHasher` it distinguishes characters that differ only in their upper byte (e.g. `'A'` `U+0041` vs `'Ł'` `U+0141`). `StringFnV1Hasher.Hash(s)` equals plain FNV-1 over `Encoding.Unicode.GetBytes(s)`; the empty string maps to the offset basis (`2166136261` unsigned, `-2128831035` signed), exactly as `StringFnV1AFullHasher` maps the empty string, and the dictionaries route the out-of-band `null` key around the hasher so this never collides with the empty-slot sentinel. `Hash(null)` throws `ArgumentNullException` (parameter `"key"`), matching the other string hashers. It sits at the cheap, classic end of the `string` escalation ladder as a peer of the FNV-1a variants; prefer `StringFnV1AFullHasher` for general use and escalate to the throughput-oriented strong family for clustered or adversarial keys. Incremental progress on issue #24.

- `StringDjb2Hasher` in `Celerity.Hashing` — Daniel J. Bernstein's classic **djb2** hash provider for `string` keys, over the full little-endian UTF-16 byte stream (both bytes of every character). Struct hasher, `[MethodImpl(MethodImplOptions.AggressiveInlining)]`. It is the simplest and cheapest of the classic string hashes: the accumulator is seeded with the magic constant `5381` and each byte is folded with the single step `hash = hash * 33 + b`, where `* 33` lowers to a shift-and-add (`(hash << 5) + hash`) — so the function uses no real multiply, no table, and no finalizer, strictly cheaper than even FNV-1a's per-byte prime multiply. Like the other full-width string hashers it folds both bytes of every 16-bit code unit (low byte then high byte), exactly djb2 over the string's native little-endian UTF-16 byte stream, so it distinguishes characters that differ only in their upper byte (e.g. `'A'` `U+0041` vs `'Ł'` `U+0141`, which `StringFnV1AHasher` collides on). Its tradeoff is weaker avalanche — djb2 has no shift/xor diffusion step, so a single changed input byte propagates less thoroughly than under `StringJenkinsOaatHasher` — but it is famous, fast, and distributes well on the short ASCII identifiers that dominate many real key sets. It sits at the cheapest, classic end of the `string` escalation ladder, a peer to the FNV-1a variants and `StringJenkinsOaatHasher`: `StringDjb2Hasher` (cheapest, shift-and-add, classic, weaker avalanche) / `StringFnV1AHasher` (low-byte only) → `StringFnV1AFullHasher` / `StringFnV1A64Hasher` (cheap FNV-1a, full Unicode width) → `StringJenkinsOaatHasher` (cheap, full Unicode width, multiply-free, stronger per-bit avalanche) → the throughput-oriented strong family. `StringDjb2Hasher.Hash(s)` equals plain djb2 over `Encoding.Unicode.GetBytes(s)`; the empty string maps to the seed constant `5381` (no characters are folded), exactly as FNV-1a maps the empty string to its offset basis, and the dictionaries route the out-of-band `null` key around the hasher so this never collides with the empty-slot sentinel. `Hash(null)` throws `ArgumentNullException` (parameter `"key"`), matching the other string hashers. Prefer it when you want the simplest, most familiar cheap hash on short ASCII identifiers; step up to `StringJenkinsOaatHasher` (still cheap, but with proper shift/xor diffusion) or the FNV-1a variants when djb2's weak avalanche clusters keys, and escalate to the throughput-oriented strong family for clustered or adversarial keys. Incremental progress on issue #24.

- `UInt64WangNaiveHasher` in `Celerity.Hashing` — an extremely cheap XOR-fold hash provider for `ulong` keys, the `ulong` counterpart to `Int64WangNaiveHasher`. Struct hasher, `[MethodImpl(MethodImplOptions.AggressiveInlining)]`. It fills the cheap-default tier of the `ulong` escalation ladder, which until now jumped straight from `UInt64WangHasher` (the full Thomas-Wang `hash64shift` finalizer) to `UInt64Hasher` (the Murmur3 `fmix64` finalizer) with no cheap XOR-fold option — even though `int` (`Int32WangNaiveHasher`), `long` (`Int64WangNaiveHasher`), and `uint` (`UInt32Hasher`) all ship one. The formula is `(int)key ^ (int)(key >> 32) ^ (int)(key >> 16)`: two ops vs the seven-step shift-add Wang mixer and the two-multiply `fmix64`. The extra middle-16-bits fold over a naive `key.GetHashCode()` keeps a chunk of the high-half entropy in the result, which materially improves distribution on keys whose low 32 bits are sequential (e.g. monotonically allocated IDs whose upper bits carry type / shard). Because the fold runs over the raw bits, `UInt64WangNaiveHasher.Hash(u)` returns exactly `Int64WangNaiveHasher.Hash((long)u)` for any matching 64-bit pattern. The XOR-fold maps `0 → 0`; the dictionaries route the out-of-band zero key around the hasher so this never collides with the empty-slot sentinel. Prefer it over `UInt64Hasher` / `UInt64WangHasher` when the key distribution is already reasonably uniform and latency matters more than collision resistance; escalate to `UInt64WangHasher` (full Wang finalizer) or `UInt64Hasher` (`fmix64`) for adversarial or heavily clustered keys. `UInt64WangHasher`'s XML remarks now cross-reference it as the cheaper drop-down tier. Incremental progress on issue #24.

- `StringFnV1AFullHasher` in `Celerity.Hashing` — a full-character-width FNV-1a 32-bit hash provider for `string` keys, the full-UTF-16 counterpart to `StringFnV1AHasher`. Struct hasher, `[MethodImpl(MethodImplOptions.AggressiveInlining)]`. It directly answers the long-standing note in `StringFnV1AHasher`'s own remarks — "for keys with significant non-ASCII content, a full UTF-8 or UTF-16 hash is preferable" — which until now had no in-box answer at the FNV-1a cost class: `StringFnV1AHasher` folds only the low byte of each character (`c & 0xFF`) and therefore collides characters that differ only in their upper byte (e.g. `'A'` `U+0041` and `'Ł'` `U+0141`), and the only escalation was straight to `StringMurmur3Hasher` (block reads, rotates, and the `fmix32` finalizer). The new hasher folds both bytes of every 16-bit code unit, low byte then high byte — exactly the FNV-1a of the string's native little-endian UTF-16 byte stream (`StringFnV1AFullHasher.Hash(s)` equals plain FNV-1a over `Encoding.Unicode.GetBytes(s)`) — so it keeps upper-byte-distinct characters separate while still costing only a pair of XOR / multiply steps per character. It fills the middle tier of the `string` escalation ladder: `StringFnV1AHasher` (cheapest, low-byte only) → `StringFnV1AFullHasher` (cheap, full Unicode width) → `StringMurmur3Hasher` (strongest avalanche). The empty string maps to the FNV-1a offset basis (`2166136261` unsigned, `-2128831035` signed), exactly as `StringFnV1AHasher` does; the dictionaries route the out-of-band `null` key around the hasher so this never collides with the empty-slot sentinel. `Hash(null)` throws `ArgumentNullException` (parameter `"key"`), matching the other string hashers. Prefer it over `StringFnV1AHasher` whenever keys contain non-ASCII characters the low-byte fold would collide; escalate to `StringMurmur3Hasher` when FNV-1a's weaker avalanche pushes clustered or adversarial keys into long probe chains. Incremental progress on issue #24.

- `StringFnV1A64Hasher` in `Celerity.Hashing` — a full-character-width FNV-1a **64-bit** hash provider for `string` keys, the wide-accumulator counterpart to `StringFnV1AFullHasher`. Struct hasher, `[MethodImpl(MethodImplOptions.AggressiveInlining)]`. It folds the same little-endian UTF-16 byte stream as `StringFnV1AFullHasher` (low byte then high byte of every 16-bit code unit) but accumulates into a 64-bit state using the FNV-1a 64-bit parameters (offset basis `14695981039346656037`, prime `1099511628211`), then xor-folds the result to a signed 32-bit value with the standard FNV retraction `h ^ (h >> 32)`. Carrying twice as many bits through the accumulation means intermediate values collide far less often, so for longer keys and larger key sets the distribution holds up better than the 32-bit FNV-1a hashers before the final fold, while still costing only a pair of XOR / multiply steps per character — and each multiply is 64-bit, which on 64-bit platforms is the same throughput as a 32-bit multiply, so the wider state is effectively free there. It extends the `string` escalation ladder with a fourth rung: `StringFnV1AHasher` (cheapest, low-byte only) → `StringFnV1AFullHasher` (cheap, full Unicode width, 32-bit state) → `StringFnV1A64Hasher` (full Unicode width, 64-bit state) → `StringMurmur3Hasher` (strongest avalanche). `StringFnV1A64Hasher.Hash(s)` equals plain FNV-1a-64 over `Encoding.Unicode.GetBytes(s)` xor-folded to 32 bits; the empty string maps to the 64-bit offset basis folded to 32 bits, and the dictionaries route the out-of-band `null` key around the hasher so this never collides with the empty-slot sentinel. `Hash(null)` throws `ArgumentNullException` (parameter `"key"`), matching the other string hashers. `StringFnV1AFullHasher`'s XML remarks now cross-reference it as the wider-state step-up. Prefer it over `StringFnV1AFullHasher` when keys are long or numerous enough that the 32-bit accumulator starts clustering; escalate to `StringMurmur3Hasher` when FNV-1a's weaker per-bit avalanche pushes clustered or adversarial keys into long probe chains regardless of state width. Incremental progress on issue #24.

- `StringXxHash32Hasher` in `Celerity.Hashing` — a throughput-oriented strong-distribution hash provider for `string` keys using the xxHash32 (XXH32) algorithm, seed `0`, over the string's native little-endian UTF-16 byte stream. Struct hasher, `[MethodImpl(MethodImplOptions.AggressiveInlining)]`. This is the first member of the XXHash family called for in issue #24, and the throughput-oriented sibling of `StringMurmur3Hasher` at the strong-avalanche top of the `string` ladder: `StringFnV1AHasher` (cheapest, low-byte only) → `StringFnV1AFullHasher` (cheap, full Unicode width, 32-bit state) → `StringFnV1A64Hasher` (full Unicode width, 64-bit state) → `StringMurmur3Hasher` / `StringXxHash32Hasher` (strong avalanche). xxHash processes the input in 16-byte stripes (eight UTF-16 code units) across **four** independent accumulators advanced by `acc = rotl(acc + lane * PRIME32_2, 13) * PRIME32_1`, merges them via `rotl(v1,1) + rotl(v2,7) + rotl(v3,12) + rotl(v4,18)`, mixes in the byte length and any remaining 4-byte lanes / trailing 2-byte char, and finishes with the XXH32 avalanche so every input bit reaches every output bit. Keeping four accumulators in flight (vs MurmurHash3's single accumulator over one 32-bit block at a time) is what lets it pull ahead on throughput as keys get longer; for very short keys MurmurHash3's simpler loop has less fixed overhead. Like the other full-width string hashers it consumes the full 16-bit value of every character, so it distinguishes characters that differ only in their upper byte (e.g. `'A'` `U+0041` vs `'Ł'` `U+0141`, which `StringFnV1AHasher` collides on). `StringXxHash32Hasher.Hash(s)` equals canonical XXH32 (seed `0`) over `Encoding.Unicode.GetBytes(s)`; the empty string maps to the well-known canonical empty-input vector `0x02CC5D05` (its UTF-16 byte stream is zero bytes), and the dictionaries route the out-of-band `null` key around the hasher so this never collides with the empty-slot sentinel. `Hash(null)` throws `ArgumentNullException` (parameter `"key"`), matching the other string hashers. `StringMurmur3Hasher`'s neighbouring docs now cross-reference it as the throughput-oriented peer. Prefer it over `StringMurmur3Hasher` when keys are long enough that the four-accumulator stripe loop wins; prefer `StringMurmur3Hasher` for very short keys. Incremental progress on issue #24.

- `StringXxHash64Hasher` in `Celerity.Hashing` — a throughput-oriented strong-distribution hash provider for `string` keys using the xxHash64 (XXH64) algorithm, seed `0`, over the string's native little-endian UTF-16 byte stream, xor-folded down to a signed 32-bit result. Struct hasher, `[MethodImpl(MethodImplOptions.AggressiveInlining)]`. It is the wide-accumulator counterpart to `StringXxHash32Hasher` (mirroring how `StringFnV1A64Hasher` widens `StringFnV1AFullHasher`) and the second member of the XXHash family called for in issue #24. Where XXH32 advances four 32-bit accumulators over 16-byte stripes, XXH64 advances four **64-bit** accumulators (`acc = rotl(acc + lane * PRIME64_2, 31) * PRIME64_1`) over **32-byte** stripes (sixteen UTF-16 code units), merges them via `rotl(v1,1) + rotl(v2,7) + rotl(v3,12) + rotl(v4,18)` followed by four merge rounds, mixes in the byte length, any remaining 8-byte / 4-byte lanes, and the trailing 2-byte char, finishes with the XXH64 avalanche (`>> 33` / `* PRIME64_2` / `>> 29` / `* PRIME64_3` / `>> 32`), and reduces the 64-bit result to 32 bits with the standard xor-fold `h ^ (h >> 32)`. Carrying twice as many bits through the accumulation keeps the distribution holding up even better than XXH32 before the fold, and on 64-bit platforms each 64-bit multiply is the same throughput as a 32-bit one while the wider stripe consumes input twice as fast — so the wider state is effectively free there and pulls ahead on longer keys. It extends the strong-avalanche top of the `string` ladder to three peers: `StringFnV1AHasher` (cheapest, low-byte only) → `StringFnV1AFullHasher` (cheap, full Unicode width, 32-bit state) → `StringFnV1A64Hasher` (full Unicode width, 64-bit state) → `StringMurmur3Hasher` / `StringXxHash32Hasher` / `StringXxHash64Hasher` (strong avalanche). - `UInt32WangHasher` in `Celerity.Hashing` — Thomas Wang's 32-bit integer hash (`hash32shift`) for `uint` keys, the `uint` counterpart to `Int32WangHasher`. Struct hasher, `[MethodImpl(MethodImplOptions.AggressiveInlining)]`. It fills the missing middle tier of the `uint` escalation ladder: the `int` family ships `Int32WangNaiveHasher` (cheap XOR-fold default) → `Int32WangHasher` (full Thomas-Wang finalizer) → `Int32Murmur3Hasher` (strongest avalanche), and `long` mirrors it, but `uint` jumped straight from the cheap XOR-fold `UInt32Hasher` to `UInt32Murmur3Hasher` with no full-Wang option in between — `UInt32Hasher`'s own remarks pointed at escalating to the Murmur3 finalizer with nothing in the middle. ### Changed

- The default hasher on the `LongDictionary<TValue>` and `LongSet` convenience subclasses switched from `Int64WangHasher` (the 11-op Thomas-Wang finalizer) to the new `Int64WangNaiveHasher` (a 2-op XOR-fold of `(int)key ^ (int)(key >>> 32) ^ (int)(key >>> 16)`). The previous default was paying roughly 5x the per-call hash cost of the naive variant on every `Hash`, and that cost compounds across the resize chain (every entry is re-hashed on resize) and across the backward-shift cluster scans (every scanned slot is re-hashed for the cyclic comparison) — the live benchmark dashboard was showing the `Long*` collections losing to BCL across every shape (1.62x on `LongDict.Insert(100k)`, 1.48x on `LongSet.Add(100k)`, 1.29-1.32x on `LongDict` / `LongSet.Remove`) almost entirely because of that. The middle-16-bits fold in the new default keeps enough high-half entropy alive that keys whose low 32 bits are sequential but upper bits carry structure (type tags, shard IDs, monotonically allocated IDs) still distribute reasonably — `Int64WangNaiveHasher` is intended as the new common-case fast path, not as a uniform replacement. **This is a user-visible behavioural change** on the parameterless `new LongDictionary<TValue>()` / `new LongSet()` constructors: callers who picked the convenience subclasses for adversarial-key collision behaviour will silently get an XOR-fold instead of the full Wang finalizer. ### Fixed

- The `this[key] = value` indexer setter on `IntDictionary<TValue, THasher>`, `LongDictionary<TValue, THasher>`, and `CelerityDictionary<TKey, TValue, THasher>` no longer calls `Resize()` on the overwrite path. Previously the setter ran its `if (_count >= _threshold) Resize();` check *before* `ProbeForInsert`, so assigning to a key that already existed while the table sat exactly at the load-factor threshold triggered a full `Resize()` — doubling the backing arrays and re-hashing every entry — even though a pure value overwrite does not grow `Count` and therefore can never push the table over its threshold. This is the indexer-setter counterpart to the `TryAdd` ordering fixed in #92: that fix reordered the five `TryAdd` implementations to probe before the threshold check, but the parallel indexer-setter path on the three dictionaries was missed, leaving the same spurious-resize anti-pattern in place (wasted `O(n)` rehash plus premature capacity growth on overwrite-heavy workloads that hover at the threshold). The fix mirrors #92 exactly: `ProbeForInsert` runs first, and `Resize()` (followed by a re-probe in the doubled table) runs only when the probe reports a genuinely new entry *and* `_count >= _threshold`. An overwrite-at-threshold now issues exactly one probe walk and leaves the backing arrays untouched; a new-key-at-threshold insert still resizes and re-probes as before. No public-API or behavioural-contract change beyond "overwrite no longer resizes" (`_version` is still bumped on every indexer set, so enumerator invalidation is unchanged). Pinned by the new `IndexerOverwriteResizeTests`, which on each of the three dictionaries fills to exactly `capacity=4` (threshold = 3) and, via a counting `IHashProvider<T>`, asserts that an overwrite-at-threshold costs exactly one `Hash` call (pre-fix it cost `threshold + 1` because the resize re-hashed every entry), that repeated overwrites never resize, and that a new key at the threshold still resizes with every entry surviving. Closes #117.

### Fixed

- `TryAdd` (and therefore `Add`) on `IntDictionary<TValue, THasher>`, `LongDictionary<TValue, THasher>`, `CelerityDictionary<TKey, TValue, THasher>`, `IntSet<THasher>`, and `CeleritySet<T, THasher>` no longer calls `Resize()` on the duplicate-key path. - `TryAdd(TKey, TValue)` on `CelerityDictionary` and `IntDictionary` — inserts without overwriting; returns `true` on success, `false` if the key already exists.  Both methods correctly handle the zero/default-key out-of-band slot.

- `Clear()` on `CelerityDictionary` and `IntDictionary` — resets the map without releasing the backing arrays, so pooled/reused instances don't pay an allocation on every generation.

### Fixed

- **`IntDictionary<TValue>` constructor arguments were silently discarded.** The convenience subclass `IntDictionary<TValue>` accepted `capacity` and `loadFactor` parameters but forwarded to `: base()` with no arguments, so every instance was created with the defaults regardless of what the caller passed. It now forwards `capacity` and `loadFactor` to the base constructor.
- **`IntDictionary` could not store the key `0`.** `EMPTY_KEY = 0` was used as the "empty slot" sentinel, which collided with the legitimate key value `0`. `map[0] = x` appeared to succeed but subsequent `ContainsKey(0)`, `map[0]`, and `Count` returned wrong answers. The zero key is now stored out-of-band via a dedicated flag + value slot, a pattern borrowed from `fastutil` / `HPPC`.
- **`CelerityDictionary` could not store `default(TKey)`.** Same root cause as above, generalized: `default(int)` / `default(long)` / `default(Guid)` / `null` strings were all lost. Fixed the same way, via a `_hasDefaultKey` flag and a dedicated value slot.

- Constructor validation test suite (`ConstructorValidationTests.cs`) covering rejection of invalid `loadFactor` (≤0, ≥1) and negative `capacity` for both `IntDictionary` and `CelerityDictionary`, plus acceptance of valid edge values.

### Fixed (additional)

- **`CelerityDictionary` and `IntDictionary` accepted invalid constructor arguments.** `loadFactor >= 1.0` caused an infinite loop in `ProbeForInsert` once the table was full; `loadFactor <= 0` caused a resize on every insert. Both constructors now throw `ArgumentOutOfRangeException` for `capacity < 0`, `loadFactor <= 0`, or `loadFactor >= 1`.

### Changed

- `TryAdd` (and therefore `Add`) on `IntDictionary<TValue, THasher>`, `CelerityDictionary<TKey, TValue, THasher>`, `IntSet<THasher>`, and `CeleritySet<T, THasher>` now walks the probe chain exactly **once** per call instead of twice. The previous implementation called `ContainsKey` / `Contains` followed by the indexer setter / `InsertNon*` helper, each starting its own probe walk; the rewrite uses a single `ProbeForInsert`-style walk that either lands on the existing entry (return `false`) or on the first empty slot (insert in place). Behaviour is identical to before — including the duplicate-key contract on `Add` and the "unchanged on duplicate" contract on `TryAdd` — but bulk-loads via the new `IEnumerable<KeyValuePair<,>>` constructor and any `Add`-heavy hot path now do roughly half the probe work. Closes issue #24. Pinned by `TryAddProbeCountTests`, which uses a counting `IHashProvider` to assert that `TryAdd` calls `Hash` exactly once on both the new-key and duplicate-key paths across all four collections.
- The `IntDictionary` `EMPTY_VALUE` field is now `static readonly` instead of an instance field. No behavior change; just removes per-instance overhead.
